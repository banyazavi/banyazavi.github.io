제 블로그에는 **Google 애널리틱스**가 설치되어 있습니다. ~~Google 애드센스도 설치되어 있으니 많이 눌러주세요~~ 이 도구를 통해 방문 통계를 볼 수 있는데요. 늘 상위권에 있는 검색 키워드가 두 개 있습니다. 하나는 **OMV5 Transmission**, 다른 하나는 **역방향 프록시**입니다.

**OMV5 트랜스미션**은 OMV4에서 플러그인으로 제공되던 트랜스미션 플러그인이 OMV5로 넘어오며 제외되었기 때문인 것 같고... **역방향 프록시**는 왜일까요? 시놀로지 DSM에서 지원되는 기능을 다른 나스에서도 적용하고 싶거나 DSM의 역방향 프록시가 잘 안되어서 찾아보는 것이 아닐까 싶습니다.

아무튼, **OMV5 트랜스미션**은 이미 글이 있어요. **Docker for a NAS** 시리즈의 [**Transmission BitTorrent 클라이언트**](/2020-03-07/Transmission-BitTorrent-클라이언트)를 보면 되고, 그 글만으로도 충분히 설정할 수 있어요. 그런데 역방향 프록시를 이야기했던 [**Nginx Proxy Manager**](/2020-03-01/Nginx-Proxy-Manager)는 조금 더 자세한 설명이 필요해 보였어요.

그래서 새 시리즈를 준비했습니다. 이 시리즈에서는 **Nginx Proxy Manager**의 모든 기능을 사용해 볼 거예요.

자세하게 어떤 기능이 가능한지는 바로 아래의 역방향 프록시 이야기에서 다룰 거예요.

# 세 번째 설명하는 역방향 프록시

[시리즈에](/2019-12-01/Nginx-역방향-프록시-설정) [한 번씩](/2020-03-01/Nginx-Proxy-Manager) 역방향 프록시에 대한 설명을 하고 있습니다. 그만큼 유용한 서비스가 아닐까 싶은데요. 이번에도 또 설명해볼게요.

![Sun Java System Web Proxy Server - 역방향 프록시 사용 - 보안 클라이언트에서 프록시 연결]({{"/assets/images/2020-09-13/2020-09-13-01-02.gif"| relative_url}})

**썬 마이크로시스템즈**의 ~~지금은 오라클~~ 프록시 웹 서버 설명서에 역방향 프록시의 개념이 잘 나와 있네요. 여기에서 우리가 사용할 전형적인 패턴에 대한 그림을 주워왔어요.

그림의 가운데에 있는 것이 역방향 프록시 서버입니다. 우리는 이 서버의 역할을 Nginx Proxy Manager가 하게 되는 것이고요, 캐시는 서비스 내에 기능으로 구현되어 있습니다. 마찬가지로 오른쪽의 클라이언트가 사용자를, 왼쪽의 서버가 연결될 서비스를 의미하게 되죠.

이처럼 역방향 프록시는 사용자와 실제 서비스의 가운데에 위치하여 연결을 중계하는 기능입니다. 엔터프라이즈 서버 운영에서는 보안성과 (실제 데이터를 가지고 있는 서버를 내부에 숨기고 역방향 프록시가 대신 노출) 부하 분산의 (같은 기능을 하는 서버를 여러 대 준비하고 역방향 프록시가 적절한 서버로 연결) 목적으로 사용하는데요.

우리는 이런 기능에 딸린 **클라이언트의 요청을 분석하여 적절한 서버로 연결**하는 것을 사용하는 거예요. 요약하자면 **포트 숫자** 기준으로 서비스에 접속하던 것을 **도메인 문자** 기준으로 접속하게 바꾸는 것이 됩니다.

예를 들어보죠. 아래와 같이 나스가 구성되어 있다고 생각해 봅시다.

> **홈 네트워크의 DDNS**: banyazavi.duckdns.org  
> **나스 내부 IP**: 192.168.0.100  
> **서비스 A의 내부 접속 주소**: http://192.168.0.100:8080  
> **서비스 B의 내부 접속 주소**: http://192.168.0.100:9090

## 포트 없이 도메인 주소만으로 연결

기존에는 이렇게 하고 있겠죠.

> **공유기의 포트 포워딩 설정**:
> - **for A**: 8080 → 192.168.0.100:8080
> - **for B**: 9090 → 192.168.0.100:9090
>
> **외부에서 다음 주소로 접속**:
> - **A**: http://banyazavi.duckdns.org:8080
> - **B**: http://banyazavi.duckdns.org:9090

그런데 포트까지 적어서 접속하는 것은 ~~간지가 안 나고~~ 외우기가 번거롭죠. 역방향 프록시를 사용하면 이렇게 접속할 수 있습니다.

> **외부에서 다음 주소로 접속**:
> - **A**: http://aaa.banyazavi.duckdns.org
> - **B**: http://bbb.banyazavi.duckdns.org

서버가 두 대 이상 있는 경우에도 유용합니다.

## 여러 서버를 도메인 주소로 연결

나스가 여러 대 있고 각각 A 서비스가 설치되어 있다고 생각해 봅시다. 내부 접속은 이렇게 되겠죠. 다른 나스의 A 서비스를 **A'**라고 할게요. 그럼,

> **A'의 내부 접속 주소**: http://192.168.0.200:8080

두 서비스를 외부에서 구분하여 접속하려면 이렇게 설정해야 합니다.

> **공유기의 포트 포워딩 설정**:
> - **for A**: 8080 → 192.168.0.100:8080
> - **for A'**: 8081 → 192.168.0.200:8080
>
> **외부에서 다음 주소로 접속**:
> - **A**: http://banyazavi.duckdns.org:8080
> - **A'**: http://banyazavi.duckdns.org:8081

외울 것이 늘어나죠. 하지만 역방향 프록시가 있으면 이렇게 하면 됩니다.

> **외부에서 다음 주소로 접속**:
> - **A**: http://aaa.banyazavi.duckdns.org
> - **A'**: http://a-dash.banyazavi.duckdns.org

## SSL 인증서를 한 곳에서 관리

HTTPS 연결을 하기 위해서 SSL을 적용할 때도 역방향 프록시에서 설정을 관리해 주면 됩니다. 모든 인증서는 역방향 프록시에서 발급받아 관리하게 되죠. 그리고 Nginx Proxy Manager는 **Let's Encrypt!**라는 무료 SSL 인증서를 매우 쉽게 발급받을 수 있는 기능을 내장하고 있어요.

## HTTPS가 지원되지 않는 서비스에 HTTPS 지원

기본적으로 HTTP는 평문 통신으로, 통신 패킷을 보면 어떤 내용을 주고받는지 바로 알 수 있습니다. 공유기 내부의 홈 네트워크에서라면 "그래 뭐... 우리 집에 해커가 있는 것도 아니고" 하며 쓰면 되지만, 외부에서 접속할 수 있게 한 경우라면 걱정이 되죠. 특히 로그인 기능이 있는 경우, 입력한 **비밀번호**가 노출될 수 있습니다!

그런데 서비스 중에서는 HTTPS 연결을 지원하지 않는 경우가 있습니다. 이런 경우에는 역방향 프록시가 대신 HTTPS 접속을 만들어 줍니다.

![Sun Java System Web Proxy Server - 역방향 프록시 사용 - 보안 클라이언트에서 프록시 연결]({{"/assets/images/2020-09-13/2020-09-13-01-02.gif"| relative_url}})

사실, 아까 썬의 설명서에서 가져온 그림이 정확히 이 기능을 설명하고 있는 거예요. **클라이언트 - 프록시**간 연결은 HTTPS를 통하여 보안 되어 접속되고, 보안 통신으로 접속한 내용을 **프록시 - 서버**간에 HTTP로 평문 통신하게 되는 구조이죠.

**프록시 - 서버**간의 평문 통신에 문제가 생기지 않냐고요? 여기는 홈 네트워크 (=내부 네트워크) 입니다. 비교적 안전한 편이죠. 그리고 엔터프라이즈급 서버 구성에서는 내부 서버가 위치한 네트워크는 오직 프록시 서버 (보통 로드 밸런서라고 칭해요. 이 기능이 더 중요하기 때문에) 만 접속할 수 있도록 구성해 놓습니다.

## 로그인 기능 지원

가끔 로그인 기능도 없는 서비스가 있습니다. 개인이 제작하여 배포하는 서비스가 이런 경우가 많은데요. 이 서비스를 외부에서도 사용하면 해킹 위험이 매우 크겠죠.

이런 상황에도 역시 역방향 프록시에서 로그인 인증을 처리하고, 인증이 된 사용자만 실제 서비스에 접속시켜 주면 되겠죠. 그리고 이 사용자 관리를 Nginx Proxy Manager에서 쉽게 할 수 있습니다.
